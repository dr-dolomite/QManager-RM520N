# Internationalization (i18n) / Language Picker

QManager ships **five bundled languages** compiled into the firmware, plus — as of v0.1.13-draft — a **runtime downloader** that installs *additional* community translation packs onto a live modem without a firmware update. The two halves are complementary: the 5 built-in languages stay bundle-only (statically imported, always present), while downloaded packs are fetched and injected at runtime. There are three linked pieces:

- **Bundle-only core** — the 5 shipped languages, picked from a switcher. See [Architecture](#architecture) and [Why the 5 core languages stay bundle-only](#why-the-5-core-languages-stay-bundle-only).
- **Contributor pipeline (Increment A)** — off-device `bun run lang` CLI to author, validate, build, and publish packs. See [Contributor pipeline (Increment A)](#contributor-pipeline-increment-a).
- **Runtime downloader (Increment B)** — the on-device CGI + worker + frontend that installs a published pack from the project's GitHub release. See [Runtime downloader (Increment B)](#runtime-downloader-increment-b).

**Status:** Increment 1 + A + B (all under v0.1.13-draft). The machinery, the three picker surfaces, and the `common`, `sidebar`, and `dashboard` namespaces are live and translated in all 5 bundled languages; the runtime downloader ships those same 3 namespaces as the pack shape. Converting the remaining component surfaces (cellular, monitoring, local-network, system-settings, onboarding, about, auth) and adding their namespaces is staged for later increments.

## Languages

Catalog in `lib/i18n/available-languages.ts` — all `bundled: true`:

| code | native | English |
|------|--------|---------|
| `en` | English | English (default / fallback) |
| `zh-CN` | 简体中文 | Simplified Chinese |
| `zh-TW` | 繁體中文 | Traditional Chinese |
| `it` | Italiano | Italian |
| `id` | Indonesia | Indonesian |

RTL is parked (all `rtl: false`): the component tree still uses physical spacing utilities (`ml-*`, `pl-*`), so an RTL language would render with broken margins. The `rtl` field + `isRtl()` helper are retained for a future logical-utilities migration.

## Architecture

Library: **`i18next` + `react-i18next`** only. Three RM551E dependencies were deliberately **not** adopted:

- **No `i18next-icu`** — ICU MessageFormat broke pluralization in the RM551E product and was reverted there. We use i18next's native `_one`/`_other` plural suffixes and `{{var}}` interpolation instead.
- **No `i18next-http-backend`** — it only exists to fetch non-bundled locales at runtime. Everything is bundled, so there is nothing to fetch; locale JSON is statically imported into `lib/i18n/resources.ts` and compiled into the JS.
- **No `i18next-browser-languagedetector`** — detection is a small hand-rolled resolver.

### The client-only-lazy invariant (load-bearing)

The app is a Next.js **static export** (`output: "export"`, no SSR). The i18next instance is created **lazily inside a `useEffect`** in `components/i18n/i18n-provider.tsx` — never at module scope and never during prerender. If `t()` were evaluated at build time, `next build` would fail. Any future i18n code must preserve this: no translation at module scope or in a server component.

### Language detection & persistence

`lib/i18n/config.ts::resolveDetectedLanguage()` resolves the initial language on cold boot:
1. `localStorage["qmanager_lang"]` (the persisted choice), else
2. `navigator.language` — exact match against the catalog, then base subtag (e.g. `en-US` → `en`), else
3. `en`.

Every `i18n.changeLanguage()` fires the provider's `languageChanged` listener, which writes `localStorage["qmanager_lang"]` and updates `document.documentElement.lang`.

## Namespaces (Increment 1)

`common` (cross-cutting chrome: actions, state, time, login, language picker), `sidebar` (nav group + item labels, keyed by a `t_key` carried on each nav item in `app-sidebar.tsx`), `dashboard` (the dashboard page + speed-test dialog + device metrics). Breadcrumbs (`hooks/use-breadcrumbs.ts`) resolve against `sidebar`.

## Adding a translatable string

1. Add the key + English value to `public/locales/en/<ns>.json` (English is the source of truth for the key set).
2. Call it from a component via `const { t } = useTranslation("<ns>")` → `t("path.to.key")`; interpolate with `t("key", { var })`.
3. Add the same key to every other locale (`zh-CN`, `zh-TW`, `it`, `id`) — translated, or English as a placeholder until a translator fills it.
4. Run `bun run i18n:check` — it fails on any missing or extra key relative to `en`, keeping all languages in parity.
5. If you add a new **namespace**, add its 5 JSON files, extend `resources.ts` (static imports + `ALL_NAMESPACES`) and the `Namespace` union in `types/i18n.ts`.

## Translations & RM551E reuse

Translations were lifted from the sibling RM551E QManager (`react-i18next`, same product) where a key exists in both — its vetted `zh-CN`/`zh-TW`/`it`/`id` values were reused verbatim. Only RM520N-specific strings (Discord Bot, Web Console, System Health Check, the persistent Data-Used counter + Storage bar, etc.) were translated fresh, anchored to the sibling repo's established terminology. Keep networking terms consistent with that anchor: RSRP/RSRQ/SINR/ARFCN/PCI/EN-DC (kept Latin), APN, IMEI, FPLMN, PDP, band locking, carrier aggregation, TTL/MTU. Brand names (Discord Bot, Tailscale, QManager) stay untranslated.

## Why the 5 core languages stay bundle-only

The 5 shipped languages are **bundled** (statically imported into the JS via `resources.ts`, always present) rather than downloaded, even though the runtime downloader now exists:

- All 5 total ~1.1 MB raw / ~274 KB gzipped — ~1% of `/usrdata`'s free space. Bundling costs a few hundred KB of firmware and guarantees the core UI is translated on a fresh install with no network round-trip.
- A bundled language can never be in a half-downloaded/failed/wiped state, so the picker for the 5 core languages is synchronous and can't fail.

The **downloader** (Increment B) handles the *different* product goal the bundle can't: *unbounded community-contributed* languages shipping faster than the firmware release cadence. Downloaded packs are additive — they extend the catalog at runtime, they do not replace the bundled 5. The two models coexist: `BUNDLED_CODES` in `available-languages.ts` is the seam that decides which path a code takes (synchronous bundle switch vs. runtime fetch + inject).

The two increments were also decoupled on purpose: the authoring pipeline (Increment A) packages and publishes community translations off-device, and the on-device downloader (Increment B) consumes them without either half reworking the pack format (the `PackMeta` / `RemoteManifest` types in `types/i18n.ts` are the shared contract).

## Contributor pipeline (Increment A)

A single zero-dependency Bun CLI, `scripts-dev/lang.ts`, wired as the `package.json` script `lang` (so it runs `bun run lang <verb>`), lets anyone author, validate, build, and publish translation packs **without touching the app or running `bun install`**. It shares one validation engine with `bun run i18n:check`, so the parity rules are identical everywhere. English (`en`) is the source of truth; the 5 current languages all validate at 100%, 0 errors. The contributor-facing walkthrough is [`docs/CONTRIBUTING-translations.md`](../CONTRIBUTING-translations.md); this section is the maintainer/architecture view.

### The shared engine — `lib/i18n/pack.ts`

`lib/i18n/pack.ts` is the one place locale JSON is loaded, flattened, and compared to English. It exports the primitives the whole pipeline is built on (`buildBaseIndex`, `compareLang`, `compareAll`, `loadNs`, `listNamespaces`, `flatten`, `pct`, and the `Issue` / `LangReport` types). `lib/i18n/check.ts` (the long-standing `bun run i18n:check` gate) was refactored to import this engine rather than re-implement parity checking — **same behavior as before, plus the new structural checks**. Because both the `i18n:check` script and the `lang` CLI call the same functions, a file that passes one passes the other.

### CLI verbs

Contributor verbs:

| Verb | What it does |
|------|--------------|
| `lang scaffold <code> [--native-name …] [--english-name …] [--rtl]` | Creates `public/locales/<code>/` with English-shaped skeleton files where every value is an empty string `""`, plus a `_pack.json` metadata stub (`{code, native_name, english_name, rtl, contributors:[]}`). Rejects invalid BCP-47 codes and existing dirs. Empty `""` means "not translated yet" — the app falls back to English and `build` strips empties. **Does not** register the language in `available-languages.ts` / `resources.ts` (see below). |
| `lang status [<code>] [--todo] [--ns <ns>]` | Per-namespace + overall completeness (translated/total, %). `--todo` lists every untranslated key with its English text — a work queue. No `<code>` = all languages. |
| `lang check [<code>] [--all] [--ci]` | Validates against English with **teaching** errors (file + line + English value + your value + the fix, plus a fuzzy "did you mean?" on typo'd keys). `--ci` emits machine-readable JSON on stdout (human logs go to stderr) and exits 1 on hard errors — this is what CI runs. Partial translations **pass** (English fallback); only structural problems fail. |
| `lang fmt <code>` | Re-orders a language's keys to English order + canonical 2-space/LF/trailing-newline formatting so the two diff line-for-line. Extra keys are **kept** (moved to the end) so nothing is silently dropped. |

Maintainer verbs:

| Verb | What it does |
|------|--------------|
| `lang build <code> [--version YYYY.MM.DD] [--out dir]` | Validates (**refuses** if any hard errors — there is deliberately no `--skip-check`), then packages a gzipped `ustar` tarball `qmanager-lang-<code>-<version>.tar.gz` (flat layout: `_pack.json` + namespace JSONs with empty values stripped) into `dist/lang/`, writes a `.sha256` sidecar, and prints completeness + sha256. Default version = today's date. |
| `lang publish <code> [--dry-run] [--contributors a,b] [--force-floor]` | Builds, enforces a **40% completeness floor** (refuse below unless `--force-floor`), uploads the tarball + sha256 to a persistent GitHub release tagged `language-packs`, and idempotently patches `language-packs/manifest.json` (dedupe by code, sorted). `--dry-run` prints the `gh` command + manifest entry without touching GitHub. |

Two determinism details worth knowing: the tar is written in **pure JS** (no external `tar` binary), so it builds identically on any OS and — with a fixed mtime and a `generated_at` derived from the dated version rather than wall-clock — is **byte-deterministic**, giving a reproducible sha256 a reviewer can independently verify. The `language-packs` release is created with `--latest=false` so it stays **out of the firmware/OTA update feed** (the device updater only follows the "latest" release).

### Validation policy (the gate)

The engine splits problems into **hard errors** (block merge, refuse `build`) and **warnings** (partial still ships):

| Condition | Severity |
|-----------|----------|
| Malformed JSON | Hard |
| Extra key (not in English) / extra namespace | Hard |
| Broken `{{placeholder}}` (renamed/dropped/added) | Hard |
| Mismatched or unbalanced HTML tags (`<strong>` …) | Hard |
| Missing key / empty value | Warning — partial ships, English fallback |
| Unusual plural category for the language | Warning only |

Plural handling is **CLDR-aware but union-based**: the accepted plural categories are English's ∪ the language's own (`Intl.PluralRules`). That grandfathers the existing `_one`-everywhere files while letting complex-plural languages (Polish, Russian, Arabic, …) add `_few`/`_many` — a plural-category mismatch is never a hard failure.

### CI gate

`.github/workflows/i18n-parity.yml` runs `bun run lang check --all --ci` on PRs touching `public/locales/**` (and on pushes to `main`/`development` touching `public/locales/en/**`). It posts a **sticky teaching PR comment** with a per-language completeness table + inline annotations, and **fails only on hard errors** — a partial translation is reported, not rejected. It runs in ~20–30s with no `bun install`. A translation PR template lives at `.github/PULL_REQUEST_TEMPLATE/translations.md`.

### Pack + manifest schema

The full field lists for the pack `_pack.json` and the `manifest.json` entries are documented in [`language-packs/README.md`](../../language-packs/README.md) — don't duplicate them here. In brief: the pack `_pack.json` carries `pack_format`, identity (`code`/`native_name`/`english_name`/`rtl`), `version` (YYYY.MM.DD), `app_min_version` (derived from `package.json`), `namespaces[]`, `completeness{overall, per_namespace}`, `key_count{translated,total}`, `generated_at` (derived from `version` for determinism), and `contributors[]`; each manifest entry summarizes one published pack (identity, `version`, `app_min_version`, overall `completeness`, `size_bytes`, `sha256`, `url`, `contributors[]`). The TypeScript shapes are `PackMeta` / `RemoteManifest` / `RemoteManifestEntry` in `types/i18n.ts` — forward-compat seams nothing in the app runtime imports yet.

### New languages are deliberately not auto-registered

`lang scaffold` creates the locale folder but does **not** add the language to `lib/i18n/available-languages.ts` or `resources.ts`. Registering a language into the shipped bundle stays a **maintainer/downloader decision**, on purpose: it protects the two load-bearing invariants — **bundle-only** (a catalog entry implies a statically-imported, compiled-in resource) and **client-only-lazy** static export. A contributor gets their translation to a good state; wiring it into the build is a separate, deliberate step.

## Runtime downloader (Increment B)

Increment B is the **device-side half** of RM551E parity: a user opens **System Settings → Languages**, picks a community pack from the Available list, and it downloads, verifies, installs, and becomes selectable — **without a firmware update**. It consumes the packs the contributor pipeline (Increment A) publishes to the project's GitHub release. Version stays **v0.1.13-draft** — this folds under the same draft as Increments 1 and A.

> ℹ️ NOTE: The bundled 5 languages are unaffected. Downloaded packs are additive, live on a **separate served path** (`/locales-packs/<code>/` vs. the bundled `/locales/<code>/`), and are injected at runtime rather than compiled in.

### Storage model — persistent store + re-copy (survives OTA)

The central constraint: `install_frontend` (`scripts/install_rm520n.sh`) wipes every child of the web root except `cgi-bin` on **every** OTA update, so a pack written under `www/` would vanish on the next update. The solution uses **three directories** and a **re-copy** derivation:

| Path | Owner | Role |
|------|-------|------|
| `/usrdata/qmanager/locales-packs/<code>/` | root | **Persistent store** — the source of truth, OUTSIDE `www/`, survives OTA |
| `/usrdata/qmanager/www/locales-packs/<code>/` | root | **Served copy** — what lighttpd serves; the frontend fetches `/locales-packs/<code>/<ns>.json` from here |
| `/usrdata/qmanager/locales-staging/<code>.stage` | www-data (0700) | **Staging quarantine** — where the unprivileged worker extracts + validates before the privileged commit |

Because the served copy lives under `www/`, the OTA wipe destroys it — but the persistent store survives. So `install_frontend()` gained an **idempotent re-derivation block after the wipe/copy** (`scripts/install_rm520n.sh:913-923`) that re-copies every persistent-store pack back into `www/locales-packs/` on every install/OTA run. It is safe on a fresh install (no persistent store yet) and on every subsequent update.

> ℹ️ NOTE: This is a **re-copy**, deliberately **not a symlink** (the earlier plan's first option). A symlink under `www/` would depend on lighttpd's `server.follow-symlink` being on — a contested dependency that re-copy sidesteps entirely. It is also **not a systemd boot oneshot** — the re-copy rides the installer that already runs on every OTA, so no new unit is needed. The worker writes both the persistent store and the served copy at install time, so a freshly-installed pack is immediately fetchable without waiting for the next OTA.

### Privilege boundary — the root helper

`/usrdata/qmanager/` is root-owned, so the unprivileged (`www-data`) worker **cannot** write the persistent store or the served copy. A narrow root helper does only the privileged atomic commit, mirroring the `qmanager_timezone_apply` pattern:

```
sudo -n /usr/bin/qmanager_language_pack_apply <code>        # commit
sudo -n /usr/bin/qmanager_language_pack_apply --remove <code>  # remove
```

The helper (`scripts/usr/bin/qmanager_language_pack_apply`) takes **no free-form paths** — `<code>` selects a FIXED, pre-computed staging/store path. Commit mode: re-validate the staging tree (never trust the caller), copy staging → a sibling temp dir in the persistent store then atomic-`mv` into place, re-derive the served copy from the *persistent* store (not from staging, so the two can never drift), then remove the staging quarantine. Stores stay **root-owned** — the helper never chowns them to www-data. One bare-path sudoers line authorizes it:

```
www-data ALL=(root) NOPASSWD: /usr/bin/qmanager_language_pack_apply
```

The code-safety guard (`[A-Za-z0-9-]{2,35}`, no leading/trailing/double hyphen) is **duplicated byte-for-byte** in both `qmanager_language_pack_apply` (`code_is_safe`) and `language_packs.sh` (`lp_pack_is_code_safe`) on purpose: the root helper's security boundary must not depend on sourcing a www-data-writable library file. Keep both copies in lockstep — a change to one that isn't mirrored is a silent security-boundary drift.

### CGI endpoints

Under `scripts/www/cgi-bin/quecmanager/system/language-packs/` (validate as **www-data**, never root):

| Endpoint | Method | Behavior |
|----------|--------|----------|
| `list.sh` | GET | Installed packs (from the persistent store) + the remote manifest |
| `install.sh` | POST `{code, manifest_url}` | `202 Accepted` + detached worker; `409` if an install is already running; `400` on bad/untrusted input |
| `install_status.sh` | GET | Current progress JSON (state, step, 0-100 %) |
| `install_cancel.sh` | POST | Writes a cancel sentinel the worker honors between steps |
| `remove.sh` | POST `{code}` | Removes a pack; **refuses bundled codes** (`en zh-CN zh-TW it id`) |

Shared library: `scripts/usr/lib/qmanager/language_packs.sh` (sourced by both CGI and worker). Worker: `scripts/usr/bin/qmanager_language_install`, double-forked and detached so the CGI can return `202` immediately.

### Security hardening (hard-won in validation — do not weaken)

- **Trust root = the project's own GitHub `language-packs` release, and only that.** `lp_manifest_url_is_safe()` pins every fetched URL — both the client-supplied `manifest_url` and the tarball URL read out of the manifest — to the prefix `https://github.com/dr-dolomite/QManager-RM520N/releases/download/`. This is the **SSRF gate** (Server-Side Request Forgery — tricking the server into fetching an attacker-chosen URL): without the pin, an authenticated caller could point the worker at any host. This same pin is what keeps `escapeValue:false` safe — the trust boundary is identical to OTA.
- **sha256 verified BEFORE extraction and BEFORE load.** `lp_verify_sha256` compares the downloaded tarball's digest against the manifest entry; a mismatch aborts before the archive is ever unpacked.
- **Tar extraction hardening (BusyBox 1.31.1 `tar` — weak traversal defense).** `lp_extract_tarball_safe` lists members with `tar -tvzf` (verbose, so member *type* is visible) BEFORE extracting, and rejects the whole archive if any member:
  - is a **symlink or hardlink** — detected by a ` -> ` marker in the verbose listing. This check runs **first**, because BusyBox `tar -tvzf` renders a hardlink with a leading `-` (identical to a regular file), distinguishable only by the trailing ` -> target`. A symlink named `common.json` pointing at `/etc/passwd` was a **real pre-fix bypass**: it passed the name allow-list, extracted as a live link, and would have been preserved verbatim by the root helper's `cp -r` into the world-served web root. The member-TYPE check closed it.
  - is not a **plain regular file** (column-1 `-`), or contains `..` / an absolute path, or is not in the allow-list (`_pack.json` + exactly the 3 namespace files `common`/`sidebar`/`dashboard`).
  - Size is also ceiling-checked (`LP_MAX_PACK_BYTES` = 2 MiB) against `df -P /usrdata` free space before download.
- **Concurrency lock = kernel `flock` on fd 9.** `install.sh` does `exec 9>"$LP_LOCK_FILE"` then a single non-blocking `flock -x -n 9` (409 immediately if busy). The double-forked worker **inherits** fd 9 (a shell `exec N>` fd is not close-on-exec, so it survives fork+exec), which keeps the lock held for the worker's entire run — even after `install.sh` exits and closes its own copy. The kernel auto-releases the lock if the worker dies (even `kill -9`), so there is no PID file, no staleness heuristic, and no TOCTOU window. This **replaced** an earlier `mkdir`+PID-file design that was structurally racy (proven exploitable under parallel-fired requests). It follows the `flock` precedent in `scripts/usr/bin/qcmd` and `scripts/usr/lib/qmanager/sms_alerts.sh` — a future reader should reuse that pattern, not reinvent the PID heuristic.

### Frontend — runtime injection & the cold-boot fix

Injection is `i18n.addResourceBundle(code, ns, json, true, true)` on fetched JSON (deep-merge + overwrite), inside the client-only provider — **NOT** `i18next-http-backend`. The hard invariant (no forbidden plugins, client-only-lazy static export) still holds.

**The cold-boot fallback fix.** Before Increment B, `resolveDetectedLanguage()` assumed *catalog membership ⟺ resources present*. A persisted `qmanager_lang` pointing at a downloaded (non-bundled) pack would init i18next with `lng:<code>` when that pack's resources were absent — which silently fell back to English, the worst failure mode (looks like the pack "did nothing"). The fix: `resolveInitialLanguage()` in `lib/i18n/config.ts` now returns `{ initial, pendingDownloaded }`. A stored code is accepted as `initial` **only if bundled**; a stored non-bundled code sets `initial = en` and returns the code as `pendingDownloaded`, which the provider then **async-hydrates** via the runtime loader — staying on English (with a toast) if the pack turns out to be unavailable/wiped.

The shared atomic loader `lib/i18n/runtime-pack.ts` (`loadDownloadedPack` / `switchLanguage`) fetches **all** namespaces first and only injects + `changeLanguage` if every one resolved — so the UI never half-switches into a missing-resource state. Fetches use `credentials:"same-origin"` (plain `fetch`, NOT `authFetch`) because packs are static lighttpd assets and cold-boot hydration must not trigger a 401 redirect on an unauthenticated surface.

### Known limitation — the manager UI is not itself translated

The new pack-manager copy in `language-settings.tsx` uses **English literals**, not `t()`. RM520N ships only 3 namespaces (`common`/`sidebar`/`dashboard`) with no `system-settings`/`errors` namespace to hold the keys, and adding keys would trip the 0-warning `i18n:check` parity gate. A future increment could add a `system-settings` namespace to make the manager itself translatable.
