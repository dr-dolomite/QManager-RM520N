# Internationalization (i18n) / Language Picker

QManager ships a **bundle-only** internationalization layer: every supported language is compiled into the firmware, and the user picks one from a switcher. There is **no** *runtime download-on-demand* "language pack" backend on the device — see [Why bundle-only](#why-bundle-only-no-runtime-download-backend). As of v0.1.13-draft there is, however, a **contributor pipeline** for authoring, validating, building, and publishing translation packs off-device — see [Contributor pipeline (Increment A)](#contributor-pipeline-increment-a). The on-device *downloader* that would consume those packs is still deferred (Increment B).

**Status:** Increment 1. The machinery, the three picker surfaces, and the `common`, `sidebar`, and `dashboard` namespaces are live and translated in all 5 languages. Converting the remaining component surfaces (cellular, monitoring, local-network, system-settings, onboarding, about, auth) and adding their namespaces is staged for later increments.

## Languages

Catalog in `lib/i18n/available-languages.ts` — all `bundled: true`:

| code | native | English |
|------|--------|---------|
| `en` | English | English (default / fallback) |
| `zh-CN` | 简体中文 | Simplified Chinese |
| `zh-TW` | 繁體中文 | Traditional Chinese |
| `it` | Italiano | Italian |
| `id` | Indonesia | Indonesian |

RTL is parked (all `rtl: false`): the component tree still uses physical spacing utilities (`ml-*`, `pl-*`), so an RTL language would render with broken margins. The `rtl` field + `isRtl()` helper are retained for a future logical-utilities migration.

## Architecture

Library: **`i18next` + `react-i18next`** only. Three RM551E dependencies were deliberately **not** adopted:

- **No `i18next-icu`** — ICU MessageFormat broke pluralization in the RM551E product and was reverted there. We use i18next's native `_one`/`_other` plural suffixes and `{{var}}` interpolation instead.
- **No `i18next-http-backend`** — it only exists to fetch non-bundled locales at runtime. Everything is bundled, so there is nothing to fetch; locale JSON is statically imported into `lib/i18n/resources.ts` and compiled into the JS.
- **No `i18next-browser-languagedetector`** — detection is a small hand-rolled resolver.

### The client-only-lazy invariant (load-bearing)

The app is a Next.js **static export** (`output: "export"`, no SSR). The i18next instance is created **lazily inside a `useEffect`** in `components/i18n/i18n-provider.tsx` — never at module scope and never during prerender. If `t()` were evaluated at build time, `next build` would fail. Any future i18n code must preserve this: no translation at module scope or in a server component.

### Language detection & persistence

`lib/i18n/config.ts::resolveDetectedLanguage()` resolves the initial language on cold boot:
1. `localStorage["qmanager_lang"]` (the persisted choice), else
2. `navigator.language` — exact match against the catalog, then base subtag (e.g. `en-US` → `en`), else
3. `en`.

Every `i18n.changeLanguage()` fires the provider's `languageChanged` listener, which writes `localStorage["qmanager_lang"]` and updates `document.documentElement.lang`.

> ℹ️ NOTE: `public/locales/` is the **source** path. Next.js flattens `public/` into the web root at build, so the **deployed** locale path on-device is `/usrdata/qmanager/www/locales/<lang>/<ns>.json`. (In the bundle-only model the JSON is also compiled into the JS via `resources.ts`; the on-disk copies are what a future downloader would overlay.)

## Namespaces (Increment 1)

`common` (cross-cutting chrome: actions, state, time, login, language picker), `sidebar` (nav group + item labels, keyed by a `t_key` carried on each nav item in `app-sidebar.tsx`), `dashboard` (the dashboard page + speed-test dialog + device metrics). Breadcrumbs (`hooks/use-breadcrumbs.ts`) resolve against `sidebar`.

## Adding a translatable string

1. Add the key + English value to `public/locales/en/<ns>.json` (English is the source of truth for the key set).
2. Call it from a component via `const { t } = useTranslation("<ns>")` → `t("path.to.key")`; interpolate with `t("key", { var })`.
3. Add the same key to every other locale (`zh-CN`, `zh-TW`, `it`, `id`) — translated, or English as a placeholder until a translator fills it.
4. Run `bun run i18n:check` — it fails on any missing or extra key relative to `en`, keeping all languages in parity.
5. If you add a new **namespace**, add its 5 JSON files, extend `resources.ts` (static imports + `ALL_NAMESPACES`) and the `Namespace` union in `types/i18n.ts`.

## Translations & RM551E reuse

Translations were lifted from the sibling RM551E QManager (`react-i18next`, same product) where a key exists in both — its vetted `zh-CN`/`zh-TW`/`it`/`id` values were reused verbatim. Only RM520N-specific strings (Discord Bot, Web Console, System Health Check, the persistent Data-Used counter + Storage bar, etc.) were translated fresh, anchored to the sibling repo's established terminology. Keep networking terms consistent with that anchor: RSRP/RSRQ/SINR/ARFCN/PCI/EN-DC (kept Latin), APN, IMEI, FPLMN, PDP, band locking, carrier aggregation, TTL/MTU. Brand names (Discord Bot, Tailscale, QManager) stay untranslated.

## Why bundle-only (no runtime download backend)

RM551E ships a downloadable-language-pack system (a CGI that fetches sha256-verified tarballs from GitHub releases). The **on-device runtime half** of that design was **deliberately not ported**, on evidence from the Phase 1 recon:

- All 5 languages total ~1.1 MB raw / ~274 KB gzipped — ~1% of `/usrdata`'s free space. Bundling costs a few hundred KB of firmware; the backend solves a disk problem RM520N doesn't have.
- `install_frontend` (`scripts/install_rm520n.sh`) wipes the web root except `cgi-bin` on **every** OTA update. A downloaded pack written under the web root would vanish on the next update; keeping packs elsewhere + re-linking on every OTA is permanent installer surface + regression risk.
- The pack backend is really about *unbounded community-contributed* languages shipping faster than the release cadence — a different product goal than "support these 5 known languages."

That reasoning still holds for the *runtime downloader*, which remains **deferred (Increment B)** — its device-side constraints (packs on `/usrdata` outside the web root at `/usrdata/qmanager/locales-packs/` surviving OTA, a symlink/overlay re-created every OTA, BusyBox-`tar` traversal hardening, size caps, and `i18next.addResourceBundle` injection) are documented in the port recon.

What *has* landed (v0.1.13-draft) is the **off-device authoring half** — the contributor pipeline below. It packages and publishes community translations; it does not put them on a modem. The two halves are decoupled on purpose: the community can build and validate packs today, and the downloader can be added later without reworking the pack format (the forward-compat `PackMeta` / `RemoteManifest` types in `types/i18n.ts` are the seam).

## Contributor pipeline (Increment A)

A single zero-dependency Bun CLI, `scripts-dev/lang.ts`, wired as the `package.json` script `lang` (so it runs `bun run lang <verb>`), lets anyone author, validate, build, and publish translation packs **without touching the app or running `bun install`**. It shares one validation engine with `bun run i18n:check`, so the parity rules are identical everywhere. English (`en`) is the source of truth; the 5 current languages all validate at 100%, 0 errors. The contributor-facing walkthrough is [`docs/CONTRIBUTING-translations.md`](../CONTRIBUTING-translations.md); this section is the maintainer/architecture view.

### The shared engine — `lib/i18n/pack.ts`

`lib/i18n/pack.ts` is the one place locale JSON is loaded, flattened, and compared to English. It exports the primitives the whole pipeline is built on (`buildBaseIndex`, `compareLang`, `compareAll`, `loadNs`, `listNamespaces`, `flatten`, `pct`, and the `Issue` / `LangReport` types). `lib/i18n/check.ts` (the long-standing `bun run i18n:check` gate) was refactored to import this engine rather than re-implement parity checking — **same behavior as before, plus the new structural checks**. Because both the `i18n:check` script and the `lang` CLI call the same functions, a file that passes one passes the other.

### CLI verbs

Contributor verbs:

| Verb | What it does |
|------|--------------|
| `lang scaffold <code> [--native-name …] [--english-name …] [--rtl]` | Creates `public/locales/<code>/` with English-shaped skeleton files where every value is an empty string `""`, plus a `_pack.json` metadata stub (`{code, native_name, english_name, rtl, contributors:[]}`). Rejects invalid BCP-47 codes and existing dirs. Empty `""` means "not translated yet" — the app falls back to English and `build` strips empties. **Does not** register the language in `available-languages.ts` / `resources.ts` (see below). |
| `lang status [<code>] [--todo] [--ns <ns>]` | Per-namespace + overall completeness (translated/total, %). `--todo` lists every untranslated key with its English text — a work queue. No `<code>` = all languages. |
| `lang check [<code>] [--all] [--ci]` | Validates against English with **teaching** errors (file + line + English value + your value + the fix, plus a fuzzy "did you mean?" on typo'd keys). `--ci` emits machine-readable JSON on stdout (human logs go to stderr) and exits 1 on hard errors — this is what CI runs. Partial translations **pass** (English fallback); only structural problems fail. |
| `lang fmt <code>` | Re-orders a language's keys to English order + canonical 2-space/LF/trailing-newline formatting so the two diff line-for-line. Extra keys are **kept** (moved to the end) so nothing is silently dropped. |

Maintainer verbs:

| Verb | What it does |
|------|--------------|
| `lang build <code> [--version YYYY.MM.DD] [--out dir]` | Validates (**refuses** if any hard errors — there is deliberately no `--skip-check`), then packages a gzipped `ustar` tarball `qmanager-lang-<code>-<version>.tar.gz` (flat layout: `_pack.json` + namespace JSONs with empty values stripped) into `dist/lang/`, writes a `.sha256` sidecar, and prints completeness + sha256. Default version = today's date. |
| `lang publish <code> [--dry-run] [--contributors a,b] [--force-floor]` | Builds, enforces a **40% completeness floor** (refuse below unless `--force-floor`), uploads the tarball + sha256 to a persistent GitHub release tagged `language-packs`, and idempotently patches `language-packs/manifest.json` (dedupe by code, sorted). `--dry-run` prints the `gh` command + manifest entry without touching GitHub. |

Two determinism details worth knowing: the tar is written in **pure JS** (no external `tar` binary), so it builds identically on any OS and — with a fixed mtime and a `generated_at` derived from the dated version rather than wall-clock — is **byte-deterministic**, giving a reproducible sha256 a reviewer can independently verify. The `language-packs` release is created with `--latest=false` so it stays **out of the firmware/OTA update feed** (the device updater only follows the "latest" release).

### Validation policy (the gate)

The engine splits problems into **hard errors** (block merge, refuse `build`) and **warnings** (partial still ships):

| Condition | Severity |
|-----------|----------|
| Malformed JSON | Hard |
| Extra key (not in English) / extra namespace | Hard |
| Broken `{{placeholder}}` (renamed/dropped/added) | Hard |
| Mismatched or unbalanced HTML tags (`<strong>` …) | Hard |
| Missing key / empty value | Warning — partial ships, English fallback |
| Unusual plural category for the language | Warning only |

Plural handling is **CLDR-aware but union-based**: the accepted plural categories are English's ∪ the language's own (`Intl.PluralRules`). That grandfathers the existing `_one`-everywhere files while letting complex-plural languages (Polish, Russian, Arabic, …) add `_few`/`_many` — a plural-category mismatch is never a hard failure.

### CI gate

`.github/workflows/i18n-parity.yml` runs `bun run lang check --all --ci` on PRs touching `public/locales/**` (and on pushes to `main`/`development` touching `public/locales/en/**`). It posts a **sticky teaching PR comment** with a per-language completeness table + inline annotations, and **fails only on hard errors** — a partial translation is reported, not rejected. It runs in ~20–30s with no `bun install`. A translation PR template lives at `.github/PULL_REQUEST_TEMPLATE/translations.md`.

### Pack + manifest schema

The full field lists for the pack `_pack.json` and the `manifest.json` entries are documented in [`language-packs/README.md`](../../language-packs/README.md) — don't duplicate them here. In brief: the pack `_pack.json` carries `pack_format`, identity (`code`/`native_name`/`english_name`/`rtl`), `version` (YYYY.MM.DD), `app_min_version` (derived from `package.json`), `namespaces[]`, `completeness{overall, per_namespace}`, `key_count{translated,total}`, `generated_at` (derived from `version` for determinism), and `contributors[]`; each manifest entry summarizes one published pack (identity, `version`, `app_min_version`, overall `completeness`, `size_bytes`, `sha256`, `url`, `contributors[]`). The TypeScript shapes are `PackMeta` / `RemoteManifest` / `RemoteManifestEntry` in `types/i18n.ts` — forward-compat seams nothing in the app runtime imports yet.

### New languages are deliberately not auto-registered

`lang scaffold` creates the locale folder but does **not** add the language to `lib/i18n/available-languages.ts` or `resources.ts`. Registering a language into the shipped bundle stays a **maintainer/downloader decision**, on purpose: it protects the two load-bearing invariants — **bundle-only** (a catalog entry implies a statically-imported, compiled-in resource) and **client-only-lazy** static export. A contributor gets their translation to a good state; wiring it into the build is a separate, deliberate step.
