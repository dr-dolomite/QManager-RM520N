# Internationalization (i18n) / Language Picker

QManager ships a **bundle-only** internationalization layer: every supported language is compiled into the firmware, and the user picks one from a switcher. There is **no** download-on-demand "language pack" backend ŌĆö see [Why bundle-only](#why-bundle-only-no-cgi-backend).

**Status:** Increment 1. The machinery, the three picker surfaces, and the `common`, `sidebar`, and `dashboard` namespaces are live and translated in all 5 languages. Converting the remaining component surfaces (cellular, monitoring, local-network, system-settings, onboarding, about, auth) and adding their namespaces is staged for later increments.

## Architecture

Library: **`i18next` + `react-i18next`** only. Three RM551E dependencies were deliberately **not** adopted:

- **No `i18next-icu`** ŌĆö ICU MessageFormat broke pluralization in the RM551E product and was reverted there. We use i18next's native `_one`/`_other` plural suffixes and `{{var}}` interpolation instead.
- **No `i18next-http-backend`** ŌĆö it only exists to fetch non-bundled locales at runtime. Everything is bundled, so there is nothing to fetch; locale JSON is statically imported into `lib/i18n/resources.ts` and compiled into the JS.
- **No `i18next-browser-languagedetector`** ŌĆö detection is a small hand-rolled resolver.

### The client-only-lazy invariant (load-bearing)

The app is a Next.js **static export** (`output: "export"`, no SSR). The i18next instance is created **lazily inside a `useEffect`** in `components/i18n/i18n-provider.tsx` ŌĆö never at module scope and never during prerender. If `t()` were evaluated at build time, `next build` would fail. Any future i18n code must preserve this: no translation at module scope or in a server component.

### Language detection & persistence

`lib/i18n/config.ts::resolveDetectedLanguage()` resolves the initial language on cold boot:
1. `localStorage["qmanager_lang"]` (the persisted choice), else
2. `navigator.language` ŌĆö exact match against the catalog, then base subtag (e.g. `en-US` ŌåÆ `en`), else
3. `en`.

Every `i18n.changeLanguage()` fires the provider's `languageChanged` listener, which writes `localStorage["qmanager_lang"]` and updates `document.documentElement.lang`.

## Namespaces (Increment 1)

`common` (cross-cutting chrome: actions, state, time, login, language picker), `sidebar` (nav group + item labels, keyed by a `t_key` carried on each nav item in `app-sidebar.tsx`), `dashboard` (the dashboard page + speed-test dialog + device metrics). Breadcrumbs (`hooks/use-breadcrumbs.ts`) resolve against `sidebar`.

## Adding a translatable string

1. Add the key + English value to `public/locales/en/<ns>.json` (English is the source of truth for the key set).
2. Call it from a component via `const { t } = useTranslation("<ns>")` ŌåÆ `t("path.to.key")`; interpolate with `t("key", { var })`.
3. Add the same key to every other locale (`zh-CN`, `zh-TW`, `it`, `id`) ŌĆö translated, or English as a placeholder until a translator fills it.
4. Run `bun run i18n:check` ŌĆö it fails on any missing or extra key relative to `en`, keeping all languages in parity.
5. If you add a new **namespace**, add its 5 JSON files, extend `resources.ts` (static imports + `ALL_NAMESPACES`) and the `Namespace` union in `types/i18n.ts`.

## Translations & RM551E reuse

Translations were lifted from the sibling RM551E QManager (`react-i18next`, same product) where a key exists in both ŌĆö its vetted `zh-CN`/`zh-TW`/`it`/`id` values were reused verbatim. Only RM520N-specific strings (Discord Bot, Web Console, System Health Check, the persistent Data-Used counter + Storage bar, etc.) were translated fresh, anchored to the sibling repo's established terminology. Keep networking terms consistent with that anchor: RSRP/RSRQ/SINR/ARFCN/PCI/EN-DC (kept Latin), APN, IMEI, FPLMN, PDP, band locking, carrier aggregation, TTL/MTU. Brand names (Discord Bot, Tailscale, QManager) stay untranslated.

## Why bundle-only (no CGI backend)

RM551E ships a downloadable-language-pack system (a CGI that fetches sha256-verified tarballs from GitHub releases). That design was **deliberately not ported**, on evidence from the Phase 1 recon:

- All 5 languages total ~1.1 MB raw / ~274 KB gzipped ŌĆö ~1% of `/usrdata`'s free space. Bundling costs a few hundred KB of firmware; the backend solves a disk problem RM520N doesn't have.
- `install_frontend` (`scripts/install_rm520n.sh`) wipes the web root except `cgi-bin` on **every** OTA update. A downloaded pack written under the web root would vanish on the next update; keeping packs elsewhere + re-linking on every OTA is permanent installer surface + regression risk.
- The pack backend is really about *unbounded community-contributed* languages shipping faster than the release cadence ŌĆö a different product goal than "support these 5 known languages."

If an open community-translation pipeline is ever wanted, the pack backend can be revisited; its constraints (packs on `/usrdata` outside the web root, a symlink re-created every OTA, tar-traversal hardening, size caps) are documented in the port recon.
